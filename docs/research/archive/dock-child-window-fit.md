# Dock as native child window of the Battle Tile

Research snapshot: 2026-07-21

Pinned runtime (from lockfiles in this repo):

| Package | Resolved version |
| --- | --- |
| `tauri` (crate) | **2.11.5** (`src-tauri/Cargo.lock`) |
| `tauri-runtime-wry` | **2.11.4** (`src-tauri/Cargo.lock`) |
| `tao` (transitive) | **0.35.3** (`src-tauri/Cargo.lock`) |
| `@tauri-apps/api` | **2.11.1** (`package-lock.json`) |

## Question

Can the Management Dock be attached as a **native macOS child window** of the
Battle Tile (`NSWindow addChildWindow:ordered:`) so the window server moves it
with the tile at compositor speed, instead of the throttled JS `onMoved` →
`setPosition` path in `createProductionDockWindowPort()`?

## Verdict

**Conditional go (macOS only).**

Attachment is supported at the pinned Tauri versions **without** a custom
`#[tauri::command]` and **without** a new capability permission, by passing
`parent: 'tile'` when the dock `WebviewWindow` is first created. A follow-up
implementation must not use `Webview.reparent()` on existing `WebviewWindow`
instances — that API rejects webview windows.

**Conditions before merge of runtime code:**

1. **Native observation** (`npm run tauri dev`): the tile remains reliably
   **above normal application windows** with the dock open as a child (see §3).
2. **Native observation**: `dockRect` absolute logical positions from
   `setPosition` still place the dock flush above/below the tile with the 8px
   gap after attachment (see §5).
3. **Platform scope**: enable the child path only after a **successful
   capability probe** at dock creation (see Fallback); treat Windows/Linux parent
   semantics as a separate product decision (owned / transient windows, not
   compositor-child follow).

If (1) fails, downgrade to **no go** — that outcome is valid and preserves the
tile always-on-top premise in `CONTEXT.md`.

## 1. Does the API exist for us?

**Answer:** Yes. The supported surface for two `WebviewWindow`s is the
**`parent` window option at creation time**, not post-hoc reparenting of the
dock webview.

**Evidence:** **documentation-only** (API signatures and crate source at pinned
versions); creation-time parent is the path this app would use.

### JavaScript (`@tauri-apps/api` 2.11.1)

- **`WebviewWindow` constructor** — second argument includes
  `parent?: Window | WebviewWindow | string` (window label). macOS maps this to
  `addChildWindow:ordered:` per the type docs in
  `node_modules/@tauri-apps/api/window.d.ts`.
- **`getCurrentWebview().reparent(window)`** — exists on `Webview` in
  `node_modules/@tauri-apps/api/webview.d.ts`, but **must not be used** for the
  dock: Tauri returns `Error::CannotReparentWebviewWindow` when either side is a
  `WebviewWindow` (`tauri` 2.11.5 `src/webview/mod.rs`, stable feature gate).

Production dock creation today (`src/ui/dock-window.ts`) uses
`new WebviewWindow(DOCK_WINDOW_LABEL, { … })` with no `parent`. The attach
slice would add `parent: 'tile'` (label from `src-tauri/tauri.conf.json`) on
**first** create inside `createDockWindow`.

### Rust (`tauri` 2.11.5)

- **`WebviewWindowBuilder::parent(&WebviewWindow)`** — sets parent on the
  underlying window builder; macOS uses `addChildWindow` (`src/webview/webview_window.rs`).
- No project Rust changes are required for the spike; the JS `create_webview_window`
  path already accepts `parent` in `WindowConfig`.

### Not viable for this app

- **`Webview::reparent`** after the dock exists — blocked for `WebviewWindow`.
- **Recreating the dock without `parent`** remains the only way to “detach” if
  attachment were ever reversed (see §6).

## 2. Permission and least-privilege guard

**Answer:** **No new permission** is required for `parent` on
`WebviewWindow` creation. **`accounted-permissions.json` need not change** for
attachment alone.

**Evidence:** **documentation-only**

- Existing capability `src-tauri/capabilities/default.json` already grants
  `core:webview:allow-create-webview-window` to windows `["tile", "dock"]`.
- `parent` is part of the `WindowConfig` payload for `create_webview_window`,
  not a separate ACL command.
- `core:webview:allow-reparent` is **not** in the manifest and is **not** needed
  if the implementation uses creation-time `parent` only.
- `core:webview:default` does **not** include reparent or create-webview-window
  (`tauri` 2.11.5 `permissions/webview/autogenerated/reference.md`).

`src/ui/capabilities-accounting.test.ts` would only need updating if a new
permission identifier were added to `default.json`.

### Custom `#[tauri::command]`

**Not required.** `src-tauri/src/lib.rs` has no custom commands today; this spike
does not add one. Parent attachment stays entirely in the stock Tauri webview/window
plugin surface.

## 3. What does it do to `alwaysOnTop`?

**Answer:** Both windows can still be **created** with `alwaysOnTop: true`, and
the Tao layer sets `NSFloatingWindowLevel` when that flag is set
(`tao` 0.35.3 `platform_impl/macos/window.rs`). **Whether the tile + dock stack
still wins against other applications’ windows after `addChildWindow` is not
proven here** — that is the main product risk.

**Evidence:**

| Claim | Label |
| --- | --- |
| `alwaysOnTop: true` on parent and child sets floating level at creation | **observed** — Tao spike reported `tile_aot=true dock_aot=true` (`prototype/dock-child-window-spike`) |
| Child ordering is managed relative to parent on macOS | **documentation-only** — Apple `addChildWindow` docs; Tauri `parent` platform note in `window.d.ts` |
| Tile remains above arbitrary third-party apps with child dock open | **documentation-only, unverified** — requires native observation with another app in the foreground stack |

**Implication:** If native observation shows the tile or dock dropping under
normal apps, treat that as **no go** for child attachment regardless of JS follow
latency.

## 4. Focus and activation

**Answer (expected behaviour):**

- **Clicking the dock** should activate the dock window; on macOS, child windows
  can become key; whether that **also activates the tile** is window-server
  behaviour not exercised in this spike.
- **Child key focus** — dock is created with `focus: true` today; attachment
  does not remove that option.
- **Parent hide** — Apple documents that child windows are hidden when the parent
  is ordered hidden; the tile is not hidden in normal play, but hiding the tile
  would affect the dock.

**Evidence:** **documentation-only, unverified** (Apple AppKit + Tauri window
docs). Not run in `npm run tauri dev` for this issue.

**Relation to `docs/agents/native-observation.md`:** Items “Dock window lifecycle”
and “OS close semantics” could change if the dock is a child (e.g. OS close on
tile vs dock). Re-run that checklist after implementation.

## 5. Does `setPosition` still work on an attached child?

**Answer:** **Calls succeed**, but **coordinate semantics on a child window are
ambiguous** relative to today’s absolute `dockRect` output. The follow-up must
verify Tauri `LogicalPosition` on a child `WebviewWindow` against native
geometry; do not assume screen-absolute coordinates still apply.

**Evidence:** **observed** (Tao child window, same macOS primitive Tauri uses):

From `prototype/dock-child-window-spike` (delayed reads after parent move):

```text
init tile=(1230, 560) dock=(0, -688) …
after tile+80x tile=(2620, 1120) dock=(1390, 78)
after dock set_position -50y dock=(2780, 78)
```

Interpretation:

- After the **parent** moved, the **child’s reported outer position changed**
  without a separate child move call → compositor follow is real (**observed**).
- A **child `set_outer_position`** did not behave like a simple “−50 px Y” nudge
  in screen space → **`dockRect` + `setPosition` must be revalidated under Tauri**,
  not inferred from Tao alone.

`dockRect` (`src/ui/dock-geometry.ts`) computes **monitor-aware absolute**
logical `x`/`y` (above/below from monitor midpoint). If the child API expects
parent-relative coordinates, the adapter must convert — or attachment must not
be used until absolute positioning is confirmed.

## 6. How is it detached?

**Answer:** There is **no** stock Tauri JS/Rust API to remove an macOS child
relationship without destroying the window. **Hide/show preserves attachment.**

**Evidence:**

| Claim | Label |
| --- | --- |
| `hide()` / `show()` on dock without destroying it | **observed** — Tao spike completed hide/show with positions unchanged afterward |
| Detach API (`removeChildWindow` / clear parent) | **documentation-only** — not exposed in Tauri 2.11.5 public API; teardown is `close` + recreate without `parent` |

Today’s `DockWindowPort.close()` hides; `destroy()` drops the JS handle only.
Attachment would survive hide/show cycles, matching current lifecycle.

---

## 7. Is it macOS-only?

**Answer:** **No** — `parent` is cross-platform with **different semantics**; it
is not a macOS-only no-op.

**Evidence:** **documentation-only** (`@tauri-apps/api` `window.d.ts` /
`WebviewWindowBuilder::parent` rustdoc at 2.11.5):

| OS | Behaviour |
| --- | --- |
| **macOS** | `addChildWindow:ordered:` — compositor-child follow (what this spike targets). |
| **Windows** | **Owned** window (z-order above owner; destroyed/hidden with owner minimize) — not the same as macOS child follow. |
| **Linux** | **Transient for** parent (`gtk_window_set_transient_for`) — stacking hint, not macOS child semantics. |

The fallback rule must **not** use `navigator.platform`. Use a **runtime probe**:
attempt dock creation with `parent: 'tile'` once; on `tauri://error` or a
documented failure, set an internal `dockChildAttachSupported = false` and use
the existing `onMoved` → `syncPositionFromTile` path. Optionally gate the probe
so Windows/Linux skip straight to fallback until product accepts owned/transient
semantics.

## Implementation sketch (if conditions clear)

**No `src/` changes in this issue** — sequence for a follow-up:

1. **`createDockWindow` (production adapter)** — add `parent: 'tile'` to the
   `WebviewWindow` options alongside existing flags (`alwaysOnTop`, `transparent`,
   etc.).
2. **`open`** — after `ready()`, call `show()`; **omit** registering
   `onTileMoved` → `applyPosition` when attachment probe succeeded (compositor
   follow).
3. **`reposition` / `syncPositionFromTile`** — keep for initial open placement,
   monitor-half flips, and **fallback** when attachment is unsupported; may need
   coordinate conversion if native testing shows parent-relative child positions.
4. **`close`** — unchanged (`hide()`).
5. **`destroy`** — unchanged; if the window is later recreated, repeat probe.

Custom Rust commands: **none.**

## Fallback rule (capability, not platform string)

```text
let dockChildAttachSupported = false;

async function createDockWindow(url) {
  try {
    const dock = new WebviewWindow(DOCK_WINDOW_LABEL, {
      …existing options…,
      parent: 'tile',
    });
    await waitForCreated(dock);
    dockChildAttachSupported = true;
    return wrap(dock);
  } catch {
    // recreate or use options without parent if partial create occurred
    dockChildAttachSupported = false;
    return createDockWindowFallback(url);
  }
}

// In open(): register onTileMoved only when !dockChildAttachSupported
```

Persist the flag for the lifetime of the dock webview instance (reset on
`destroy()` / failed create).
