# SideScape foundation research

Researched against the local SideScape source at commit `190e020` on 2026-07-18.

## Verdict

Use SideScape as an architectural reference and release scaffold, not as a codebase to fork wholesale. The strongest reusable ideas are its caller-pumped headless Engine, injected nondeterminism, Snapshot/event split, tolerant field-by-field save loading, pre-mount Offline Progress pump, pure window-geometry model behind native ports, asset registries with contract tests, and macOS/Windows release matrix. The party game should own new domain state, timing, save schema, animation manifests, geometry constants, and security configuration.

## Decision matrix

| Area | Reuse | Adapt | Reject or defer |
|---|---|---|---|
| Windowing | Frameless, transparent, always-on-top Tauri window; pure geometry functions; native window and webview-layout ports; serialized geometry transitions. | Replace SideScape's `320×220` compact surface and card dimensions with the fixed `480×112` Battle Tile and the management fan-out chosen by its own prototype. Persist position, but let application state own the open management layout and recompute its dimensions. | Do not copy SideScape's AppKit snapshot overlay, private API, or atomic-frame Rust until this app reproduces the same macOS compositor defect. Do not inherit the actual geometry constants or SideScape UI scale policy. |
| Headless Engine | One deep, UI-free Engine; caller-controlled advancement; injected RNG and clock; commands in, Snapshot plus discrete events out; derived legality exposed by Snapshot; invalid internal commands fail loudly. | Model a Party, continuous Action Cycles, Wave boundaries, Stage Attempts, separate persisted loot RNG, and animation projection events. The prototype must choose a finer simulation quantum or elapsed-time API instead of inheriting SideScape's 600 ms Tick. | Do not import SideScape's combat, bank, equipment, skills, activity, content, or Engine state. Do not let DOM, Tauri, timers, audio, or asset concerns enter the simulation. |
| Save | Snapshot is the persistence boundary; tolerate missing, malformed, stale, and dangling fields; inject `now`; autosave and explicit close save; keep export/import as a recovery affordance. | Add an explicit schema version and migration pipeline from the first shipped save. Validate the decoded outer shape before migration, preserve the two RNG streams and pending boundary edits, and save enough Stage state to calculate offline results before deliberately starting a fresh visible Stage Attempt. | Reject `JSON.parse(...) as Snapshot` followed by silent whole-save fallback as the long-term loader. Reject SideScape's save key, save shape, and field-level defaults. |
| Offline Progress | Compute capped elapsed work from `savedAt`; handle missing timestamps and clock skew as zero; advance the same headless simulation before UI/event subscribers mount; show one aggregate summary rather than replaying event spam. | Use the cap selected for this game and an accelerated/batched Engine entry point where profiling requires it. Apply earned XP and Drops deterministically, then discard transient combat state and begin a fresh visible Stage Attempt exactly as the product rules require. | Do not inherit SideScape's eight-hour/48,000-Tick constant, per-Tick loop as an unquestioned performance strategy, or its kill/skill/death summary vocabulary. |
| Testing | Vitest for Engine and pure UI logic; seeded RNG and injected time; tests through the public Engine boundary; Playwright browser-degraded UI coverage; Rust `cargo check`; generated-asset contract tests. | Add deterministic combat replay fixtures, save-migration fixtures, online/offline equivalence tests, Wave-boundary edit tests, fixed-size Battle Tile screenshots, native window smoke tests on macOS and Windows, and performance budgets for live effects plus accelerated offline simulation. | Do not copy SideScape's domain tests or make pixel screenshots the source of combat correctness. Browser tests cannot prove native transparency, positioning, or always-on-top behavior. |
| Asset build | Keep authored/raw sources separate from normalized runtime assets; use a registry as the source of asset identity; validate dimensions, alpha policy, palette/format rules, and source/output parity; write generated artifacts atomically; generate contact sheets for review. | Define animation clips and per-frame metadata for `32×48` Character frames, separate Ability effects, anchors, timing, direction, and provenance. Never invoke AI generation during application build or runtime. |
| Tauri | Tauri 2 + Vite + TypeScript shell; platform-neutral frontend ports; minimal Rust commands only where native behavior genuinely requires them; app icons for `.icns` and `.ico`; position/size permissions scoped to the main window. | Rename package, product, Rust crate, bundle identifier, save keys, ports, and icons. Start with `480×112`, non-resizable, decoration-free, transparent, always-on-top configuration; add only the capabilities and plugins actually used. Decide window-state restoration after the fan-out prototype. | Remove the scaffold `greet` command. Do not begin with `withGlobalTauri: true`, `csp: null`, broad opener permissions, `macOSPrivateApi`, or AppKit/WebKit dependencies unless a demonstrated requirement justifies each one. |

Pin the Node major supported by the new lockfile/toolchain rather than copying `26` blindly. | Do not retain SideScape branding or publish a production release as “unsigned personal-distribution.” Avoid creating and pushing the authoritative tag before both platform builds are known good, or provide an explicit retry/recovery procedure for failed tagged releases. |

## Primary-source evidence

### Engine and tests

- SideScape records the caller-pumped, timer-free Engine decision and its testability consequences in [ADR 0001](../../../../../sidescape/docs/adr/0001-caller-pumped-deep-engine.md). The current public boundary remains `tick()` plus commands, `snapshot()`, and typed events in [engine.ts](../../../../../sidescape/src/core/engine.ts); `createEngine` injects content, RNG, an optional Snapshot, and `now`.
- The Engine loads saved data through a tolerant field-by-field reconstruction rather than assigning the saved object directly; examples include numeric coercion, clamping, defaults, and validation of referenced equipment in [load-state.ts](../../../../../sidescape/src/core/load-state.ts).
- Core behavior is exercised through Vitest in [engine.test.ts](../../../../../sidescape/src/core/engine.test.ts). The repository also separates pure geometry tests, boot/offline tests, browser-degraded Playwright tests, and Rust checks through [package.json](../../../../../sidescape/package.json), [playwright.config.ts](../../../../../sidescape/playwright.config.ts), and [CI](../../../../../sidescape/.github/workflows/ci.yml).

### Save and Offline Progress

- SideScape's boot boundary loads a localStorage Snapshot, performs Offline Progress before mounting UI or sound subscribers, starts the caller Tick interval, and autosaves every ten seconds in [boot.ts](../../../../../sidescape/src/ui/boot.ts). Its initial parse is only a TypeScript cast and malformed JSON becomes no save, which is useful prototype simplicity but insufficient as the new game's durable migration boundary.
- Offline elapsed time is floored to Ticks, capped, and made tolerant of missing timestamps or clock skew; the same Engine is pumped while events are aggregated into one summary in [offline-progress.ts](../../../../../sidescape/src/ui/offline-progress.ts). The fixed 48,000-Tick/eight-hour cap and summary fields are SideScape decisions, not reusable requirements.
- Human-readable save export/import is isolated in [save-transfer.ts](../../../../../sidescape/src/ui/save-transfer.ts), making it a useful recovery pattern independent of the game domain.

### Windowing and Tauri

- The deployed window is non-resizable, decoration-free, always-on-top, transparent, and shadowless, but also opts into global Tauri APIs, no CSP, and macOS private API in [tauri.conf.json](../../../../../sidescape/src-tauri/tauri.conf.json). Only the visual window flags belong in the initial scaffold.
- SideScape keeps monitor clamping, anchoring, scale, and workspace capacity as Tauri-free math in [window-geometry.ts](../../../../../sidescape/src/ui/window-geometry.ts), while [window-chrome.ts](../../../../../sidescape/src/ui/window-chrome.ts) supplies native and webview ports and serializes changes. This separation is directly reusable even though none of its dimensions are.
- SideScape added custom macOS AppKit/WebKit snapshot and atomic-frame commands to mask a specific resize-compositor artifact in [lib.rs](../../../../../sidescape/src-tauri/src/lib.rs), requiring private APIs and platform dependencies declared in [Cargo.toml](../../../../../sidescape/src-tauri/Cargo.toml). These are a proven workaround for SideScape, not a default dependency for the new app.
- Window capabilities currently include drag, close, size, position, opener, and window-state permissions in [default.json](../../../../../sidescape/src-tauri/capabilities/default.json); the new app should generate the smallest list from its actual calls.

### Asset pipeline and release

- SideScape's asset scripts keep sources and normalized outputs separate, validate a declared registry, constrain dimensions/alpha/palette, and perform atomic writes in [ingest-sprite-core.mjs](../../../../../sidescape/scripts/art/ingest-sprite-core.mjs). Source/output parity and runtime-contract checks live in [sprite-assets.test.ts](../../../../../sidescape/scripts/art/sprite-assets.test.ts). The mechanics are reusable, while SideScape's square canvases and palette are not.
- The release workflow computes a semantic version from `app-v*` tags, pushes the tag, stamps placeholder versions into package/Tauri/Rust files, and builds macOS plus Windows NSIS draft-release artifacts in [release.yml](../../../../../sidescape/.github/workflows/release.yml) using [set-version.mjs](../../../../../sidescape/scripts/set-version.mjs). The matrix and stamping approach transfer cleanly; tag-before-build and unsigned distribution need an explicit project decision.

## Bootstrap recommendation for the next prototype

1. Scaffold a clean Tauri 2/Vite/TypeScript project and copy only generic tool configuration after reviewing current dependency versions.
2. Create a new `Engine` façade with injected content, combat RNG, loot RNG, and clock; keep it free of timers and browser/Tauri imports.
3. Give the prototype one caller-controlled advancement API, commands for the already-defined boundary edits, a serializable versioned Snapshot, and discrete presentation events carrying stable simulation timestamps and actor/action identities.
4. Prove the same seeded Stage produces the same Snapshot and events under live stepping, accelerated stepping, save/reload, and Offline Progress. After offline rewards are committed, prove boot starts a fresh visible Stage Attempt.
5. Keep Battle Tile window geometry behind a port and pure calculation, but postpone SideScape's advanced native resize workaround until “Prototype the compact management fan-out” establishes whether this app needs it.
6. Establish the animation asset manifest and validator separately; the Engine emits semantic combat events and never names sprite files or animation frames.

This boundary is sufficient to unblock “Prototype the simulation boundary” without importing SideScape's game domain.
